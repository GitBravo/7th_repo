# RAG 기반 병무청 보도자료 질의응답 시스템 - 작업 계획 및 의사결정 기록

## 개요
- documents/ 폴더의 병무청 보도자료 hwpx 10개를 Naive RAG 파이프라인으로 질의응답하는 웹 시스템 구축
- 임베딩: OpenAI text-embedding-3-small / 벡터DB: Supabase(pgvector) / 채팅: OpenRouter / Rerank: Cohere

## 결정 1. 문서 구조 분석
10개 문서 모두 아래 정형 템플릿을 따름 (평균 1,400자, 700~3,300자):
엠바고(선택) → 병무청/보도자료 헤더 → 보도시점/배포일 → 제목/부제 → 리드 문단 → 번호("1.")
또는 네모("□") 구분 본문 섹션(하위 "ㅇ" 항목, "기존→개선" 비교 박스 포함) → 담당부서/연락처 표(전 문서 공통 보일러플레이트)

## 결정 2. 청킹 전략 - 구조 인식 청킹 (Structure-aware chunking)
**선택**: 고정 길이 분할 대신 번호/네모 헤더 단위로 섹션을 나눠 청킹.
**이유**: 문서가 짧고(평균 1,400자) 섹션 경계가 명확해, 고정 길이 분할 시 하나의 정책 항목이
두 청크로 잘리거나 서로 다른 항목이 한 청크에 섞여 검색 정확도가 떨어질 위험이 큼.
**세부 규칙**:
- 헤더(`\d{1,2}\. ` 또는 `□`) 등장 전 리드 문단(제목/부제/요약)은 별도 청크로 분리
- 헤더가 전혀 없는 문서(1건, 예: 고용노동부 업무협약)는 문서 전체를 1개 청크로 유지
- 섹션이 800자 초과 시 문장 경계 기준 추가 분할 + 약 80자 오버랩 (실제로는 발생 사례 없음, 최대 784자)
- 임베딩 입력(`embed_text`)에는 `[문서 제목] (보도일자)`를 프리픽스로 추가해 청크 단독으로도 문맥 파악 가능하게 함.
  단, DB의 `content` 컬럼에는 프리픽스 없는 원문만 저장 (표시/인용 시 깨끗한 원문 유지 목적)
- 결과: 10개 문서 → 총 38개 청크, 길이 분포 80~784자(평균 294자)

## 결정 3. 보일러플레이트(담당부서 연락처 표) 처리
**선택**: 임베딩 대상에서 완전히 제외 (DB에 적재하지 않음).
**이유**: 모든 문서 끝에 반복되는 담당자/연락처 표는 내용 질의응답에 기여하지 않고 노이즈만 추가함.
**구현**: "담당 부서"로 시작하는 문단을 만나면 그 이후 전체를 버림 (`strip_boilerplate`).
추가로 "병무청/보도자료/보도시점/배포/배포이후/엠바고 문구/순수 날짜 문단" 등 서식 라벨도 노이즈로 판단해 제거.

## 결정 4. Supabase 테이블/함수명
**선택**: `news_chunks` 테이블 / `match_news_chunks` 함수.
**이유**: 기존 Supabase 프로젝트에 이전 실습 흔적인 `documents`/`documents_test` 테이블과
`match_documents_test` 함수가 이미 존재함을 발견 (이번 병무청 문서와 무관). 지침 8번(테이블/함수명 겹치지 않게 구성)에 따라 새 이름 사용.
**스키마**: LangChain SupabaseVectorStore 표준과 유사 (`id, content, metadata jsonb, embedding vector(1536)`).
SQL 파일: `sql/setup_supabase.sql`

## 결정 7. RAG 파이프라인 / 웹 UI 설계
**웹 프레임워크**: Streamlit (사용자 선택 - 과제/프로토타입에 적합, 채팅 UI 내장 컴포넌트로 개발 속도 빠름)
**채팅 모델**: OpenRouter `openai/gpt-4o-mini` (사용자 선택 - 저렴하고 한국어 품질 양호)
**Rerank 모델**: Cohere `rerank-multilingual-v3.0` (한국어 지원 다국어 rerank 모델)
**파이프라인 (`rag_pipeline.py`)**: 질의 임베딩(OpenAI) → `match_news_chunks` RPC로 top-10 후보 검색
→ Cohere rerank로 top-4 재정렬 → 컨텍스트 구성 후 OpenRouter 채팅 모델로 답변 생성 (naive RAG: 단일 단계 검색, 계층/멀티홉 없음)
**웹 UI (`app.py`)**: Streamlit 채팅 인터페이스, 답변과 함께 참고 자료(문서 제목/날짜/섹션) 표시

## 결정 8. 종단 테스트 완료
- 사용자가 `.env`에 `OPENROUTER_API_KEY`, `COHERE_API_KEY` 입력 완료 (Cohere는 트라이얼 키 - 소규모 테스트 용도로 충분, rate limit만 유의)
- `pip install -r requirements.txt`로 cohere/streamlit 설치
- Python 직접 호출로 파이프라인 스모크 테스트: "사회복무요원 관련 새로운 지원 제도가 있어?" 질의 → 서로 다른 2개 문서(업무협약, 하반기 제도변경)에서 관련 내용을 정확히 종합한 답변 확인
- `streamlit run app.py`로 실제 서버 구동 후 헤드리스 브라우저(Playwright)로 채팅 입력 → 답변 + 참고자료 expander 렌더링 확인, 콘솔 에러 없음

## 결정 9. 실시간 자동 임베딩 기능 추가
**요구사항**: documents/ 폴더에 파일을 추가하면 실시간으로 자동 청킹+임베딩+Supabase 적재.
**구현**:
- 청킹 로직(`chunk_documents.py`)과 임베딩/적재 로직(`embed_and_upload.py`)을 각각
  `hwpx_chunker.py` / `supabase_loader.py` 공용 모듈로 분리 (배치 스크립트와 실시간 감시가 동일 로직 재사용)
- `watch_documents.py`: `watchdog` 라이브러리로 documents/ 폴더의 파일 생성(`on_created`)/이동(`on_moved`,
  다운로드 시 임시파일→rename 패턴 대응)을 감지. 파일 크기가 일정 시간 변화 없을 때까지 대기(`_wait_until_stable`)한 뒤
  복사 도중 처리하는 것을 방지하고 나서 청킹+임베딩+적재 수행
- **멱등성(idempotent) 처리**: 같은 `source_file`의 기존 청크를 삭제 후 재삽입(`delete_chunks_by_source`)하도록
  `supabase_loader.embed_and_load_chunks`를 설계 → 워처 재시작/파일 재처리 시에도 중복 행 발생하지 않음
- FastAPI 서버(`backend.py`) 기동 시 `watch_documents.start_watcher()`를 자동 실행 → 앱이 떠 있는 동안 실시간 반영
- 독립 실행도 가능: `python watch_documents.py` (웹 서버 없이 감시만 필요할 때)
**검증**: 기존 문서를 복사해 새 파일명으로 documents/에 추가 → 수 초 내 자동 감지·청킹·임베딩·적재 확인(5개 청크),
이후 파일 및 테스트 데이터 삭제로 정리, 전체 row 수가 38개로 원복됨을 확인.

## 결정 10. 웹 프레임워크 전환 - Streamlit → FastAPI + Tailwind CSS
**배경**: 사용자가 UI 디자인에 Tailwind CSS 적용을 요청. Streamlit은 자체 React 컴포넌트를 내부적으로
렌더링하는 구조라 Tailwind 유틸리티 클래스를 표준적으로 적용하기 어려움 (사용자에게 트레이드오프 설명 후 확인받음).
**결정**: Streamlit(`app.py`) 제거, FastAPI(`backend.py`) + 정적 HTML/JS(`static/index.html`, `static/app.js`) +
Tailwind CDN 조합으로 교체. `rag_pipeline.py`(검색/rerank/생성 로직)는 전혀 수정하지 않고 그대로 재사용 —
`POST /api/chat` 엔드포인트로 감싸기만 함. 즉 "기능은 그대로, 디자인/프레젠테이션 레이어만 교체".
**디자인**: 다크 테마 채팅 UI, 사용자/어시스턴트 말풍선, 답변 하단에 접이식 "참고 자료" 패널(출처 문서명/날짜/섹션/발췌).
**검증**: `uvicorn backend:app`로 서버 구동 → 헤드리스 브라우저(Playwright)로 실제 질의 → 답변 및 참고자료 4건
정상 렌더링 확인 (콘솔 에러 없음, favicon 404는 무해한 브라우저 기본 요청).
**부수 효과**: `requirements.txt`에서 `streamlit` 제거, `watchdog`/`fastapi`/`uvicorn` 추가.

## 결정 11. 채팅 말풍선 줄바꿈 버그 수정
**문제**: 말풍선에 `whitespace-pre-wrap` 클래스를 적용한 상태에서, 템플릿 리터럴의 들여쓰기/줄바꿈
문자가 그대로 텍스트 노드에 포함되어 실제 내용 위아래에 빈 줄처럼 보이는 버그 발생.
**해결**: `innerHTML` 템플릿 문자열 대신 `textContent`로 답변/질문 텍스트를 직접 대입하도록 변경
(`renderUserMessage`, `buildRagCard`의 답변 영역). 구조적 HTML(출처 카드, 확장 질의 칩)은 별도 엘리먼트로
분리해 pre-wrap 컨테이너 밖에 둠으로써 의도한 줄바꿈(LLM 답변의 문단 구분)만 유지되도록 함.

## 결정 12. Advanced RAG 추가 - Naive RAG와 답변 비교
**요구사항**: 사용자가 질문하면 Naive RAG와 Advanced RAG 두 파이프라인의 답변을 동시에 비교해서 보여줄 것.
**Advanced RAG 기법 선택 - Multi-Query Expansion**:
- 이유: (1) 추가 인프라 없이 기존 Supabase/임베딩 구조 위에 바로 얹을 수 있음 (하이브리드 검색처럼 새
  컬럼/인덱스 필요 없음), (2) "검색 단계"에서만 차이를 줘서 두 파이프라인의 rerank/생성 로직을 동일하게
  유지 → 비교 시 "질의 확장이 검색 품질에 주는 영향"만 순수하게 드러남, (3) 개념 설명이 쉬워 UI에 그대로
  노출(확장 질의 칩)해도 사용자가 이해하기 쉬움.
- **동작**: LLM(OpenRouter `gpt-4o-mini`)이 원 질문을 다른 키워드/관점의 검색 질의 3개로 확장
  (`generate_query_variants`) → 원본 질문 포함 총 4개 질의로 각각 벡터 검색(top-6) 후 청크 id 기준
  병합/중복 제거(유사도 높은 것 우선) → 이후 rerank(top-4)·답변 생성은 Naive RAG와 동일 로직 재사용.
- **Naive RAG(기존)는 그대로 유지**: 원 질문 1개로만 벡터 검색(top-10) → rerank(top-4) → 생성.
- `rag_pipeline.py`: `run_naive_rag`, `run_advanced_rag`, `compare_rag` 함수로 정리.
- `backend.py`: `/api/chat` 응답을 `{naive: {...}, advanced: {...}}` 구조로 변경 (한 번의 질문 요청으로
  두 파이프라인 결과를 함께 반환).
- **UI**: 질문 1개당 카드 2개(NAIVE RAG / ADVANCED RAG)를 나란히 표시 (`md:grid-cols-2`, 모바일은 1열로 스택).
  Advanced RAG 카드에는 확장 질의를 칩(chip) 형태로 노출해 어떤 검색어들이 추가로 쓰였는지 투명하게 보여줌.
**참고**: CLAUDE.md의 "RAG 방식은 Naive RAG 방식을 사용" 제약은 baseline 파이프라인 정의로 유지하고,
Advanced RAG는 비교 목적의 별도 파이프라인으로 추가한 것 — 필요 시 CLAUDE.md에도 반영 가능(사용자 요청 시 추가 예정).
**검증**: 헤드리스 브라우저로 데스크톱/모바일 두 화면에서 테스트, 두 카드 모두 정상 답변+출처 렌더링,
콘솔 에러 없음(favicon 404 제외), 말풍선 줄바꿈 버그 재확인 결과 정상.

## 결정 13. 좌측 사이드바 대화 이력 + 새 대화 버튼
**요구사항**: 좌측 사이드 메뉴에 대화 이력을 자동으로 남기고, 이력을 선택해 기존 대화를 이어갈 수 있게 하고,
우측 상단에 새 대화 생성 아이콘 추가.
**저장 방식**: 서버 DB가 아닌 브라우저 `localStorage`에 저장 (`vmc_chat_state_v1` 키).
**이유**: 로그인/사용자 계정 개념이 없는 단일 사용자 데모 앱이라, 서버 측에 대화를 저장하려면
사용자 구분(세션/계정) 개념을 새로 도입해야 해서 배보다 배꼽이 커짐. localStorage는 별도 인프라 없이
브라우저별로 이력이 자동 유지되고, 대화 데이터(질문+양쪽 RAG 답변+출처+확장질의) 전체를 그대로 캐시해두므로
같은 대화를 다시 열 때 API를 재호출하지 않고 즉시 재생(replay)됨.
**동작**:
- 대화는 사용자가 첫 질문을 보낼 때 자동 생성됨(제목 = 첫 질문 앞부분). 빈 "새 대화" 상태로는 이력에 남기지 않음.
- 사이드바 항목은 `updatedAt` 내림차순(최근 대화가 위로) 정렬, 활성 대화는 에메랄드 강조 스타일로 표시.
- 항목 클릭 시 저장된 메시지(사용자 질문 + naive/advanced 비교 카드)를 그대로 재생 - **RAG는 매번 새로
  질의응답하는 것이지 이전 대화 맥락을 검색/생성에 반영하는 멀티턴 메모리는 아님** (요청 범위가 "이력 UI"였고
  대화형 문맥 유지는 별도 기능이라 이번 스코프에는 포함하지 않음).
- 항목에 마우스오버 시 삭제(×) 버튼 노출 (요청엔 없었지만 이력 목록의 표준 기능이라 추가; 삭제 시 로컬에서만 제거).
- 새로고침해도 `activeId`를 함께 저장해두어 마지막으로 보던 대화가 유지됨.
- 새 대화 버튼(우측 상단 + 아이콘): 활성 대화를 해제하고 빈 화면으로 전환 (다음 질문 전송 시 새 대화로 생성됨).
**반응형**: 데스크톱은 사이드바 항상 표시, 모바일은 기본적으로 숨겨진 오프캔버스 드로어 + 헤더의 햄버거
버튼으로 열고, 배경 클릭 또는 × 버튼으로 닫음.
**검증**: 헤드리스 브라우저로 (1) 질문 2개로 대화 2개 자동 생성 및 정렬 확인, (2) 사이드바에서 이전 대화 선택 시
정확히 재생됨, (3) 새로고침 후에도 마지막 대화 유지, (4) 모바일 뷰포트에서 사이드바 열기/닫기 정상 동작 확인.
